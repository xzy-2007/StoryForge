# StoryForge 项目过程文档（SPEC_PROCESS）

项目名称：StoryForge —— 可视化剧情节点编辑器

版本：v2.0

---

## 1. 文档目的

本文档记录 StoryForge 项目从最初想法到形成最终 SPEC 与 PLAN 的过程，以及冷启动验证的结果。

---

## 2. 项目想法形成过程

### 2.1 初始想法

项目开始阶段，我考虑过多个方向：

- 学习助手
- 娱乐性问卷平台
- 角色设计工具
- 世界观管理工具
- 剧情管理工具

最初希望制作一个具有 AI 元素的产品，但经过分析发现：

1. AI 并不是项目必须条件
2. 如果强行加入 AI，容易导致项目变成简单的 API 调用封装
3. 本课程更关注软件工程流程，而非 AI 功能数量

因此最终决定选择偏向工程工具的剧情节点编辑器。

### 2.2 项目定位演变

```
角色卡工具 → 剧情管理工具 → 可视化剧情节点编辑器
```

决定转向节点编辑器的原因：角色卡工具核心交互较弱，与普通表单工具区别有限；而节点编辑器具有可视化编辑、图结构管理、节点交互等更明显的软件工程特点。

## 3. Brainstorming 关键节点记录

### 3.1 第一轮：确定项目方向

**AI 追问的问题：**
- "你的项目真正解决什么问题？"
- "用户是谁？核心价值是什么？"
- "哪些功能是必须的，哪些只是扩展？"

**我的回应：** 最初倾向"角色设计卡"工具。

**AI 的反问：** "角色卡工具与普通表单工具的区别在哪里？"

**我的调整：** 经过分析，角色卡确实交互太弱。转向剧情节点编辑器，因为具有更丰富的交互模式和数据结构。

**决策：** 制作类似 Unity 编辑器的剧情设计工具。

### 3.2 第二轮：明确项目范围

**AI 提出的扩展建议：**
- AI 自动生成剧情
- 角色数据库
- 世界观管理
- 剧情分析

**我的决定：拒绝。**

**理由：** 这些功能会快速扩大项目范围，且：
- AI 功能不是核心，且容易变成简单的 API 调用
- 数据结构复杂化
- 难以保证最终完成度

**最终确定：** 项目只关注"剧情节点的可视化编辑"。

### 3.3 第三轮：确定核心模块

最终确定 5 个核心模块：

1. 剧情节点管理
2. 节点连接管理
3. 可视化编辑区域
4. 节点属性编辑
5. 项目保存与加载

同时明确暂不实现：AI 生成、角色卡、世界观数据库、游戏运行。

## 4. SPEC 设计迭代

### 4.1 数据模型设计

**初始设计包含 `note` 字段：**

```typescript
interface Node {
  id: string
  type: NodeType
  title: string
  description: string
  position: Position
  note: string  // 初始存在
}
```

**决策：删除 `note` 字段。**

理由：note 不是核心需求，保留会导致属性编辑器增加字段、测试增加、数据结构复杂化。

### 4.2 节点类型表示方式

**初始：** 未明确说明使用数字枚举还是字符串枚举。

**冷启动中发现问题：** 新 Agent 无法判断使用哪种形式。

**决策：** 明确使用字符串枚举。

```typescript
enum NodeType {
  EVENT = "EVENT",
  DIALOGUE = "DIALOGUE",
  CHOICE = "CHOICE",
  END = "END"
}
```

理由：JSON 可读性更强，方便人工查看项目文件。

### 4.3 数据创建方式

**冷启动中 Agent 提出：** PLAN 中只描述字段，没有规定对象创建方式。

**决策：** 提供 `createNode()` 和 `createEdge()` 工厂函数，自动生成 ID、初始化默认值。

## 5. 冷启动验证

### 5.1 验证目的

根据课程要求，使用一个**与主开发智能体不同类型**的 agent，在**不提供任何历史对话**的前提下，仅凭 `SPEC.md` + `PLAN.md` 尝试实现 1–2 个 task。

### 5.2 操作步骤

```
1. 启动一个全新的 Agent session（不同智能体类型）
2. 不导入任何先前会话或记忆
3. 仅提供 SPEC.md 和 PLAN.md 文件
4. 指定从 PLAN 中选择 Task 1 和 Task 2
5. 要求："遇到不确定之处即暂停询问，而非凭猜测继续"
```

### 5.3 冷启动发现的问题

#### 问题 1：Node 字段定义不一致

**发现：** 功能描述中存在 `note` 字段，但数据模型和 PLAN 中没有。

**影响：** 未来实现属性编辑器时可能产生歧义。

**修改：** 删除 `note` 字段。

#### 问题 2：NodeType 表示方式不明确

**发现：** SPEC 没有说明枚举形式。Agent 无法判断使用数字枚举还是字符串枚举。

**修改：** 明确使用字符串枚举。

#### 问题 3：创建函数是否需要定义

**发现：** SPEC 只定义数据结构，没有说明创建方式。Agent 自主选择使用 `createNode()` / `createEdge()` 工厂函数。

**评估：** 该方案有利于保证数据一致性，因此接受。

## 6. AI 建议与人工决策总结

| AI 建议 | 是否接受 | 原因 |
|---|---|---|
| 增加 AI 剧情生成 | 否 | 扩大范围，不符合核心目标 |
| 增加角色系统 | 否 | 复杂度过高 |
| 增加世界观管理 | 否 | 超出编辑器核心 |
| 删除 note 字段 | 是 | 减少歧义 |
| NodeType 使用字符串 | 是 | 提高可读性 |
| 使用创建函数 | 是 | 提高数据一致性 |
| 使用 React Flow | 是 | 降低节点编辑器开发难度 |
| 增加主题系统 | 是 | 提升用户体验 |
| 增加导出功能 | 是 | 增加实用价值 |

## 7. 对 Brainstorming 技能的评价

### 做得好的地方

- 帮助项目从模糊想法逐渐收敛到具体方向
- 强迫明确用户、目标和范围
- 提醒避免功能无限扩展
- 通过追问帮助发现设计中的模糊点

### 不足之处

- AI 有时会倾向提出大量扩展功能（AI 生成、世界观系统等），这些功能虽有吸引力但不一定适合课程项目
- 最终仍需要人工判断：哪些功能真正提升项目价值，哪些是范围蔓延

---

## 8. 总结

通过 brainstorming 和冷启动验证，项目从一个较宽泛的"剧情设计工具"逐渐明确为：**一个专注于剧情节点可视化编辑的桌面软件**。

冷启动过程证明：初版 SPEC 已经能够指导基本开发，但仍存在部分隐藏假设。经过修改后，数据模型更加明确，功能范围更加清晰。

本过程体现了 AI4SE 中的重要观点：**AI 可以帮助生成方案和代码，但工程师仍需要负责定义问题、判断需求和控制系统边界。**
