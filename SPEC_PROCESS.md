# 2. 项目想法形成过程


## 2.1 初始想法

项目开始阶段，我考虑过多个方向，包括：

- 学习助手；
- 娱乐性问卷平台；
- 角色设计工具；
- 世界观管理工具；
- 剧情管理工具。


最初希望制作一个具有 AI 元素的软件，但经过分析发现：

1. AI 并不是项目必须条件；
2. 如果强行加入 AI，容易导致项目变成简单的 API 调用；
3. 本课程更关注软件工程流程，而不是 AI 功能数量。


因此最终决定选择一个更加偏工程工具的软件。

# 3. Brainstorming 过程记录


## 3.1 第一次关键讨论：确定项目方向


### AI 提出的问题

AI 引导我思考：

- 项目真正解决什么问题？
- 用户是谁？
- 核心价值是什么？
- 哪些功能是必须的，哪些只是扩展？


### 我的原始想法

最初倾向制作：

“角色设计卡”或者“世界观管理工具”。


### 调整原因

进一步分析后发现：

角色卡工具虽然容易实现，但核心交互较弱，与普通表单工具区别有限。


相比之下，剧情节点编辑器具有更明显的软件特点：

- 可视化编辑；
- 图结构管理；
- 节点交互；
- 数据组织。


因此确定：

制作类似 Unity 编辑器的剧情设计工具。


---

## 3.2 第二次关键讨论：明确项目范围


### AI 提出的建议

AI 建议可以加入：

- AI 自动生成剧情；
- 角色数据库；
- 世界观管理；
- 剧情分析。


### 我的决定

拒绝加入。


原因：

这些功能会导致项目范围快速扩大，并且：

- AI功能不是核心；
- 数据结构复杂；
- 难以保证最终完成度。


最终确定：

项目只关注：

“剧情节点的可视化编辑”。


---

## 3.3 第三次关键讨论：确定核心功能


经过讨论，最终确定核心模块：


1. 剧情节点管理；

2. 节点连接管理；

3. 可视化编辑区域；

4. 节点属性编辑；

5. 项目保存与加载。


同时明确：

暂不实现：

- AI自动生成剧情；
- 角色卡；
- 世界观数据库；
- 游戏运行。

# 4. SPEC 设计过程


## 4.1 数据模型设计


初始设计包含：

Node：

- id
- type
- title
- description
- position
- note


经过分析发现：

note 字段虽然可以提供额外信息，但不是核心需求。


如果保留：

会导致：

- 属性编辑器需要增加字段；
- 测试增加；
- 数据结构复杂。


因此删除 note。


最终 Node：

```text
id

type

title

description

position
4.2 节点类型设计

初始设计没有明确规定：

NodeType 使用数字还是字符串。

在冷启动测试中，新的 Agent 提出疑问：

如果使用数字枚举：

{
"type":1
}

项目文件可读性较差。

因此修改：

NodeType 使用字符串枚举：

enum NodeType {
 EVENT="EVENT",
 DIALOGUE="DIALOGUE",
 CHOICE="CHOICE",
 END="END"
}

原因：

JSON 文件更加直观；
方便人工查看；
降低维护成本。
4.3 数据创建方式

冷启动 Agent 提出：

PLAN 中只描述字段，没有规定对象创建方式。

经过讨论后决定：

允许提供：

createNode()

createEdge()

作用：

自动生成 id；
初始化默认字段；
保证数据结构一致。

这不是强制业务功能，而是实现层面的辅助设计。

8. 冷启动发现的问题
问题1：Node 字段定义不一致

发现：

功能描述中存在 note 字段。

但是：

数据模型和 PLAN 中没有。

影响：

未来实现属性编辑器时可能产生歧义。

修改：

删除 note 字段。

问题2：NodeType 表示方式不明确

发现：

SPEC 没有说明枚举形式。

Agent 无法判断：

数字枚举还是字符串枚举。

修改：

明确使用字符串枚举。

问题3：创建函数是否需要定义

发现：

SPEC 只定义数据结构，没有说明创建方式。

Agent 自主选择：

createNode()

createEdge()

经过评估：

该方案有利于保证数据一致性，因此接受。

9. AI 建议与人工决策总结

删除 note 字段	是	减少歧义
NodeType 使用字符串	是	提高可读性
使用创建函数	是	提高数据一致性
10. 对 brainstorming 技能的评价
做得好的地方
帮助项目从模糊想法逐渐收敛；
强迫明确用户、目标和范围；
提醒避免功能无限扩展。
不足

AI 有时会倾向提出大量扩展功能。

例如：

AI生成剧情；
世界观系统；
角色数据库。

这些功能虽然有吸引力，但不一定适合课程项目。

最终仍需要人工判断：

哪些功能真正提升项目价值。

11. 总结

通过 brainstorming 和冷启动验证，项目从一个较宽泛的“剧情设计工具”逐渐明确为：

一个专注于剧情节点可视化编辑的软件。

冷启动过程证明：

初版 SPEC 已经能够指导基本开发，但仍存在部分隐藏假设。

经过修改后：

数据模型更加明确；
功能范围更加清晰；
后续 Agent 更容易按照要求开发。

本过程体现了 AI4SE 中的重要观点：

AI 可以帮助生成方案和代码，但工程师仍需要负责定义问题、判断需求和控制系统边界。
