# REFLECTION —— StoryForge 项目开发反思

## 一、Superpowers 工作流体验

### 1.1 哪些技能发挥了最大作用？

在 StoryForge 项目中，Superpowers 最大的价值是帮助我建立了一套相对规范的软件开发流程。其中对我影响最大的几个技能是 `writing-plans`、`test-driven-development` 和 `requesting-code-review`。

项目开始时，我对于 StoryForge 的想法比较模糊，只知道想做一个故事流程编辑工具，但并没有明确功能边界。通过 writing-plans，我将需求拆分成多个 Task，每个 Task 都有明确目标、涉及文件和验证方式，让原本比较大的想法变成了可以逐步实现的小目标。

TDD 和 Code Review 则帮助我控制 AI 生成代码的质量。因为 AI 写代码速度很快，如果缺少约束，很容易产生一些“看起来完成了，但实际上不符合需求”的实现。通过测试和 review，我们可以更容易发现问题。

### 1.2 哪些部分“形式大于实质”？

实际开发过程中，完整执行 Red → Green → Review 流程确实增加了一些时间成本。对于一些简单修改，例如 UI 文字调整，完整流程有时显得比较重。另外，Review 中发现的一些 Minor 问题并不会影响当前功能运行。如果每个问题都立即修复，会影响开发节奏。

---

## 二、TDD 在 AI 协作下的体验

### 2.1 TDD 是阻碍还是放大器？

在这个项目中，我认为 TDD 更多是一种放大器。

如果直接让 AI 实现“故事编辑器”，它可能会加入很多当前不需要的功能。而通过提前设计测试，例如“节点可以显示标题”“节点之间可以连接”“数据可以保存”，AI 的实现范围会更加明确。

Task4 的保存加载功能就是一个例子。最初需求只是“实现保存”，如果没有测试约束，AI 可能只会增加一个保存按钮。但通过测试要求 saveProject 必须保存节点和边的数据，loadProject 必须恢复状态，最终实现才真正满足需求。

当然，TDD 也带来了额外工作量。但在 AI 协作环境下，代码生成速度很快，错误扩散也更快，因此测试实际上成为了限制 AI 偏离方向的重要手段。

### 2.2 是否出现过偏离 TDD 的情况？

在部分 UI 调整阶段，我曾经倾向于让 AI 直接修改代码，再补测试。这样短期效率较高，但容易遗漏交互细节。

后来我发现，只要功能涉及状态变化或者组件之间的关系，就应该先明确测试目标，否则 AI 很容易实现一个“视觉上正确”的版本，而不是符合实际需求的版本。

## 三、Subagent-Driven 工作流

### 3.1 智能体能自主运行多久？

在任务目标明确的情况下，Subagent 可以完成一个完整的小功能。例如 Task2 和 Task3 中，从创建测试、实现功能到运行验证，基本能够自主推进。

但如果任务描述比较宽泛，例如“优化编辑器”，AI 容易根据自己的理解扩展功能。因此，AI 能自主工作的时间，很大程度取决于任务拆分是否清晰。

### 3.2 什么样的 Task 颗粒度最好？

我认为一个 Task 最合适的是完成一个完整的小功能闭环。

任务过大会导致 AI 同时处理多个问题，例如数据结构、UI 和状态管理，最终代码容易混乱。任务过小则会增加大量沟通和流程成本。

在实践中，一个 Task 能够独立完成、独立测试、独立 review，是比较合适的粒度。

## 四、SPEC 与 PLAN 的影响

### 4.1 规约不清导致的问题

项目中比较明显的问题出现在保存加载功能。

最初我只描述了“实现项目保存和加载”，但没有明确说明 StoryForge 是 Electron 桌面应用，以及未来希望支持真实文件保存。因此 AI 默认选择了 localStorage 方案。

虽然这个方案在 Web 应用中合理，但和桌面编辑器的目标存在差距。这让我意识到，AI 不会自动理解隐藏需求。如果 SPEC 没有说明运行环境、数据生命周期和用户场景，AI 会选择它认为合理的方案，而不是我真正想要的方案。

### 4.2 冷启动验证的价值

冷启动验证体现了 SPEC 的重要性。

如果一个新的智能体只能依靠 SPEC 和 PLAN 开发，而没有之前的聊天上下文，那么文档必须足够清晰。这个过程让我认识到，SPEC 不只是记录设计，它也是团队之间传递信息的重要工具。

## 五、Prompt / Context 策略

### 5.1 最有效的 Prompt 方法

在使用 OpenCode 的过程中，我发现最有效的方法不是简单描述目标，而是给出明确限制。例如，“实现保存功能”通常会得到比较宽泛的结果，而“先设计测试目标，只实现 ProjectManager 序列化，不修改 UI”会让 AI 更容易理解当前阶段。同时，告诉 AI 当前已经完成的功能、不能修改的部分，也能够减少无关修改。

### 5.2 为什么有效？

大模型具有很强的代码生成能力，但不会自动拥有完整项目目标。清晰的 Prompt 相当于增加了一层约束，让 AI 在正确范围内完成任务。

## 六、凭据与分发

### 6.1 工程要求带来的思考

虽然 StoryForge 最终没有加入 LLM 调用，但课程中关于凭据安全和分发的要求让我意识到，一个完整的软件不仅是代码能够运行，还需要考虑安全、安装和使用流程。过去做项目时，我通常只关注“程序能不能跑”，但真正的软件还需要考虑别人如何获取、配置和使用。

### 6.2 分发过程中的挑战

Electron 项目让我认识到，开发环境运行成功并不代表软件完成。

`npm run dev` 可以启动程序，但真正交付还需要考虑打包、平台兼容和依赖管理。这部分也是我之前较少接触的工程内容。

---

## 七、如果重新开始

### 7.1 我会改变什么？

如果重新开始，我会更早确定项目范围。

最初 StoryForge 的设想包含 AI 辅助创作、复杂编辑等功能，但实际开发后发现，仅完成一个稳定的故事编辑器已经包含很多工程问题。

缩小范围、优先完成核心功能，可能会让最终项目质量更高。

### 7.2 对 Superpowers 的批判

Superpowers 假设开发者有足够时间执行完整流程，也假设 AI 能在明确任务下稳定工作。

实际项目中，第一个假设并不完全成立，因为课程时间有限，需要根据任务重要程度调整流程。

第二个假设基本成立，但前提是任务拆分和需求描述足够清楚。如果输入模糊，AI 仍然会按照自己的理解实现。

---

## 八、总结

通过 StoryForge 项目，我更加理解了 AI 时代软件工程师的价值。

AI 可以快速生成代码，但它不能替代工程师判断需求、设计架构和验证结果。真正重要的能力不是写更多代码，而是提出正确的问题、建立合理约束，并判断最终结果是否真正解决问题。

Superpowers 提供了一套很好的开发框架，但最终决定项目质量的仍然是人的判断。未来的软件工程师可能更多承担系统设计者和质量控制者的角色，而不仅仅是代码编写者。
